Zooko's Triangle

In 2001, Zooko Wilcox-O'Hairn observed a fundamental trilemma for naming systems. He argued that a name can have at most two of these three properties:

DNS (.com domains)
✅ Human-memorable: example.com
✅ Secure: TLS certificates
❌ Centralized: ICANN controls everything
Nostr NPUBs
✅ Decentralized: anyone generates keys
✅ Secure: 128-bit secp256k1
❌ Not human-memorable: npub1q3x8z...
Our question: Can we push NPUBs closer to the center of the triangle — adding human-readability without sacrificing security or decentralization? Vanity mining is one approach. But it has a problem...

Understanding the Metrics

While we mine, here's how the scanner will evaluate each candidate. We use three complementary metrics: unique character count (diversity — how many distinct characters appear), max frequency (raindrop — how many times the most common character repeats), and Shannon entropy (distribution — how uneven the character distribution is). Combined quality = max(rarity_unique, rarity_max, rarity_entropy) — the best of any metric.

Unique Characters: The Diversity Metric

The unique character count measures how many distinct bech32 characters appear in a window. Fewer unique chars = more recognizable "low-diversity" pattern.

rarity_unique = expectedUnique − uniqueChars (higher = rarer = better)

A 16-character window with only 3 unique characters looks like:

[p][p][q][p]
[p][q][q][p]
[q][p][p][q]
[p][q][p][p]

A 16-character window with 10 unique characters looks like:

[7][k][2][x]
[m][v][0][z]
[3][r][5][k]
[8][j][q][n]

Same window size. But one is instantly recognizable and the other looks like noise. Fewer unique characters = bigger blocks of the same color = more recognizable to humans.

Occupancy problem — expected unique chars in random windows:
For a window of W chars from a 32-char alphabet, E[unique] = 32 × (1 − (31/32)^W)
W=16: ~10.3 unique (out of 16 possible) → rarity = 10.3 − actual
W=25: ~16.6 unique
W=36: ~20.7 unique
W=49: ~24.2 unique

rarity > 0 = fewer unique than expected = recognizable pattern
rarity = 0 = exactly as expected = normal random noise
rarity < 0 = more unique than expected (not useful for fingerprinting)

Max Frequency: The Raindrop Metric

The max frequency measures how many times the most common character repeats in a window. A big "raindrop" of one character is visually striking.

rarity_max = max_freq − expected_max_freq (higher = rarer = better)

For W=16: E[max_freq] ≈ 3.08 (from Monte Carlo simulation, 5M samples). A window where one character appears 8 times is a big visual cluster — max_freq=8, rarity_max = 8 − 3.08 = 4.92.

"a3a2aaaa5a7aa9a" → max_freq=8 (char 'a' appears 8 times) → rarity_max = 4.92 → very recognizable
"7da6b6a3eb923ab0" → max_freq=3 (normal) → rarity_max ≈ 0 → looks like noise

rarity_max > 0 = bigger raindrop than expected = recognizable
rarity_max ≈ 0 = normal random (max_freq ≈ 3 for W=16)

Max frequency catches "one big raindrop" patterns: a window like "aaaaaaaaa1234567" has max_freq=9 (one char dominates) even though unique=8 looks normal. Diversity alone would miss this.

Shannon Entropy: The Distribution Metric

How uneven is the character distribution? Random windows have H≈3.2 bits. Lower = more uneven = more recognizable. Catches patterns like 'aaabbcdeeff12345' where no single char dominates but the distribution is still skewed.

rarity_entropy = expected_entropy − shannonEntropy(window) (higher = rarer = better)
Expected Shannon entropy for random windows (32-char alphabet):
W=16: H≈3.22 bits (below log₂(32)=5 due to sampling collisions)
W=25: H≈3.85 bits
W=36: H≈4.20 bits
W=49: H≈4.45 bits

"aaabbcdeeff12345" → H≈3.0 bits → rarity_entropy = 3.22 − 3.0 = 0.22 → slightly uneven
"aaaaaaa1aaaaaaa2" → H≈1.0 bits → rarity_entropy = 3.22 − 1.0 = 2.22 → very uneven

rarity_entropy > 0 = more uneven than expected = recognizable distribution
rarity_entropy ≈ 0 = normal random distribution

Entropy catches "skewed distribution" patterns: a window where no single char dominates but several chars are overrepresented. Diversity and max_freq might both look normal, but entropy still flags the unevenness.

Why Three? Maximum Degrees of Freedom

Each metric catches different patterns. OR logic means a window qualifies if ANY metric is exceptional. This maximizes Alice's chances of finding a recognizable pattern while the attacker must still replicate the EXACT pattern.

max_freq catches what unique misses:
"aaaaaaaaa1234567" → max_freq=9, unique=8 — diversity looks normal (8 ≈ 10.3) but one char dominates → rarity_max=5.92

unique catches what max_freq misses:
"aaaaabbbbbccccc6" → max_freq=5, unique=4 — max is moderate (5 > 3.08) but only 4 colors → rarity_unique=6.3

entropy catches what both miss:
"aaabbcdeeff12345" → unique=9, max_freq=3 — both look normal, but distribution is skewed → H≈3.0, rarity_entropy=0.22

Together they catch ALL THREE types of patterns.
Combined quality = max(rarity_unique, rarity_max, rarity_entropy) — whichever metric is most extreme wins.

The asymmetry: Alice needs ANY recognizable pattern. An attacker needs ALICE'S SPECIFIC pattern. More dimensions of recognizability = more degrees of freedom for Alice = harder for attacker to replicate any specific one.

The scanner ranks windows by combined quality descending — highest max(rarity_unique, rarity_max, rarity_entropy) wins. Shannon entropy is also displayed as a secondary metric showing the evenness of the distribution.

The Asymmetry

Alice's fingerprint (left) was discovered for free. An attacker (right) generates a new key with the same vanity prefix — but their fingerprint is completely different. This is the core of the asymmetric vanity approach.

The core argument:

Alice needs ANY recognizable pattern. Every key she generates might contain one — a low-diversity window, a big raindrop, or a skewed distribution. With three metrics and many window positions, she has thousands of chances per key. Finding a pattern is like finding any raindrop in a storm — easy, because they're everywhere.

An attacker needs ALICE'S SPECIFIC pattern. They can't just find any pattern — they must replicate the exact one Alice has: same characters, same positions, same window size, same metric profile. Finding a specific pattern is like finding that one specific raindrop — astronomically harder.

This is the asymmetry: Alice's cost = find any raindrop. Attacker's cost = find that specific raindrop.

More dimensions of recognizability = more degrees of freedom for Alice = harder for attacker. Each metric (unique chars, max frequency, Shannon entropy) adds another way for Alice to find a pattern. But the attacker must match ALL dimensions of Alice's specific pattern simultaneously. More metrics don't make it harder for Alice — they make it harder for the attacker.
Key insight: The attacker's key has the same vanity prefix but a completely different fingerprint. They would need to grind millions of keys — each one scanned for Alice's exact pattern — and the cost grows exponentially with the number of metrics. This is why the asymmetry works: Alice finds any pattern for free, the attacker must replicate one specific pattern at astronomical cost.
