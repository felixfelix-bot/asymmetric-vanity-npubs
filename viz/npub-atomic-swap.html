Atomic Sale of NPUB Offset

How to sell the grinding offset without trusting either party

What is Atomic About an Atomic Swap?

Atomic = both sides happen or neither does. No one can cheat.

The trick: use a hash lock.

Alice has a secret (the offset d). She hashes it: h = SHA256(d).
Bob locks payment: "whoever reveals a value x where SHA256(x) = h gets the money."

To claim the money, Alice MUST publish d. The hash guarantees she can't fake it.
Once d is public, Bob reads it. Both sides complete simultaneously.

If Alice walks away: Bob gets a refund after a timeout. Nobody loses.

The Core Trick Applied to NPUB Offsets

The offset d is a small integer (~5 bytes). Perfect preimage for an HTLC.

Seller has found d such that (user_pub + d*G) = desired vanity NPUB
Buyer wants d to compute new_nsec = (old_nsec + d) mod n

d is the perfect atomic swap asset:
✓ Small (fits in any hash preimage)
✓ Verifiable (buyer can check resulting NPUB before paying)
✓ Useless to anyone except the buyer (need original nsec)
✓ Hash-lockable (standard SHA256 works)

Option 1: Bitcoin On-Chain HTLC

TRUE ATOMIC EXPENSIVE
1
Seller finds offset d, computes target NPUB, sends Buyer: target_npub + h = SHA256(d)
2
Buyer verifies: target_npub contains "meshmate"? Has high z-score? (can verify WITHOUT knowing d)
3
Buyer creates Bitcoin script: "pay seller IF they reveal preimage of h, OR refund buyer after 48h"
4
Seller claims BTC by broadcasting d on-chain. d is now public in the witness data.
5
Buyer reads d from blockchain → computes new_nsec → verifies match. Done.
Pros: Battle-tested. No trust needed. Works with any Bitcoin wallet.
Cons: On-chain fee $1-10. May exceed the value of the vanity service itself.
Speed: ~10 minutes for Alice's claim to confirm.

Option 2: Lightning HTLC

TRUE ATOMIC NEEDS CHANNELS
1
Same setup: seller sends target_npub + h
2
Buyer sends Lightning payment with payment_hash = h
3
Seller claims payment by providing preimage d
4
Buyer reads d from Lightning invoice settlement
Pros: Near-zero fees ($0.001). Instant settlement.
Cons: Both need Lightning channels + inbound liquidity. Must be online.
Speed: Seconds.

Option 3: Cashu + NUT-11 HTLC (BEST for our use case)

TRUE ATOMIC MICROPAYMENT PRIVATE NOSTR-NATIVE
Cashu NUT-11 defines spending conditions including HTLC locks for ecash tokens.
1
Seller finds offset d, sends Buyer: target_npub + h = SHA256(d)
2
Buyer verifies target NPUB. Then mints a Cashu token locked with NUT-11 HTLC condition: hashlock = h
3
Seller claims ecash from Mint by providing preimage d. Mint verifies H(d) = h.
4
d is now known to the mint (and visible in the claim). Buyer reads d → computes new_nsec.
Pros: Zero fees. Instant. Private (ecash). Fits the Nostr ecosystem. Micropayment-friendly.
Cons: Requires mint to implement NUT-11 HTLC (not all mints do yet). Mint learns d (but d is useless without the buyer's nsec).
Speed: Instant.

Option 4: Simple Cashu (No HTLC) — Good Enough for V1

NOT ATOMIC REPUTATION-BASED
1
Seller shows target_npub (proves it exists without revealing d)
2
Buyer sends Cashu token
3
Seller sends offset d
4
Buyer verifies: new_nsec produces target_npub? ✓
Pros: Dead simple. Works TODAY with any Cashu mint. No special features needed.
Cons: Seller could take the money and not send d. Buyer trusts seller's reputation.
Mitigation: Seller stakes reputation. For small amounts (few hundred sats), risk is minimal.
For demo: This is fine. Upgrade to NUT-11 later.

Option 5: Adaptor Signatures — Why It Doesn't Work Here

DOESN'T WORK
Adaptor signatures reveal a point (d*G), not the scalar (d).

Bob could learn d*G → compute new npub → verify identity.
But Bob CANNOT compute new_nsec from d*G alone (discrete log problem).

So adaptor sigs give Bob the ability to verify but not to use the key.
Would need a second channel to transmit the actual scalar. Not truly one-shot.

Recommended Path

V1 Demo: Simple Cashu (pay → receive offset → verify). Fast to build, easy to demo.

V2 Production: Cashu NUT-11 HTLC. True atomic. The seller can't run off with the money.

V3 Optional: Add Lightning fallback for non-Cashu users.

On-chain Bitcoin is overkill for amounts measured in hundreds of sats. The fee would exceed the service price. But the mechanism is identical — just change the payment rail.
